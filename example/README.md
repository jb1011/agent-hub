# Skill Hub SDK examples

This folder shows how to consume the local SDK from `backend/sdk`.

The scripts import `../backend/sdk/dist/index.js` directly, so you must build the SDK before running them:

```bash
npm run build --prefix backend/sdk
```

Also install the example folder dependencies:

```bash
npm install --prefix example
```

In another terminal, start the backend:

```bash
npm run dev --prefix backend
```

For `register:provider`, the backend must know the registry contract via `AGENT_HUB_REGISTRY_ADDRESS`, or via `AGENT_HUB_CHAIN_ID` / `ESCROW_CHAIN_ID` with a deployment file available. It must also be able to sign the EIP-712 `RegisterProviderAuthorization` with `DELIVERY_ATTESTER_PRIVATE_KEY` (same address as `AgentHubConfig.deliveryAttester()` on-chain).

To automatically link `registry_provider_id` after `registerProvider`, start the backend with `ARC_RPC_WS_URL` (`ProviderRegistered` listener). Without a websocket, you can still set `registry_provider_id` manually via `PATCH /providers/:request_id`.

If you want the example to sign and send transactions, also set `RPC_URL` and `SIGNER_WALLET_PK` in `example/.env`. The key must match the signer expected by the transaction: `owner_wallet` for register provider, `user_wallet` for create job / accept job, `signer_wallet` for start job.

You can then run the examples from the repo root:

```bash
npm run health --prefix example
npm run providers:list --prefix example
npm run register:provider --prefix example
npm run create:job --prefix example
npm run start:job --prefix example
npm run accept:job --prefix example
npm run refund:queue --prefix example -- <jobId>
npm run refund:final --prefix example -- <jobId>
npm run worker --prefix example
```

## Provider agent + Skill Hub (recommended)

When a user submits a job on the Skill Hub UI, the frontend calls your registered `api_base_url/chat` with:

```json
{
  "message": "user prompt",
  "job_request_id": "0x…",
  "skillhub_api_url": "http://159.223.137.183:3000"
}
```

Wire `example/agent-skillhub-chat-handler.js` into `agent-poet` `/chat` so that when `job_request_id` is present you run start-job → your LLM → job-finish (env: `SKILLHUB_API_URL`, `SIGNER_WALLET_PK`). No `PROVIDER_REQUEST_ID` polling needed.

### Optional: poll worker

If you cannot change `/chat`, run `provider-worker.ts` beside `agent-poete`:

1. Poll `GET /jobs?status=FUNDED&provider_request_id=<bytes32>`
2. `start-job` (sign with `SIGNER_WALLET_PK` = provider `signer_wallet`)
3. `POST` `CHAT_URL` with `{"message":"<job.input.uri>"}`
4. `job-finish` with `output` matching your provider `output_schema` (plain string if schema `type: "string"`)

```bash
cp example/.env.example example/.env
# API_URL=http://159.223.137.183:3000
# PROVIDER_REQUEST_ID=0x37d4...   (from /jobs/0x37d4... page URL)
# SIGNER_WALLET_PK=...
# CHAT_URL=http://127.0.0.1:3000/chat

npm run build --prefix backend/sdk
npm run worker:once --prefix example
pm2 start "npm run worker --prefix /path/to/skill-hub/example" --name skillhub-worker
```

`/chat` must receive valid JSON (`JSON.stringify({ message })`). Smart quotes or `{ message: "x" }` cause `SyntaxError` in agent-poete logs.

## Configuration

The scripts only read `API_URL`, `RPC_URL`, and `SIGNER_WALLET_PK` from the environment. You can start from `.env.example`:

```bash
cp example/.env.example example/.env
```

Registration payloads live in:

```text
example/config/provider.json
example/config/job.json
example/config/start-job.json
example/config/acceptance.json
```

Executable scripts are in:

```text
example/scripts/
```

The scripts automatically load `example/.env` when run via `npm --prefix example`.

## Provider identifiers

The API distinguishes two identifiers:

| Field | Format | Usage |
|-------|--------|-------|
| `request_id` | `bytes32` (e.g. `0xabc…`) | API key: `GET/PATCH/DELETE /providers/:id`, returned by `POST /providers` |
| `registry_provider_id` | decimal integer (e.g. `"1"`) | On-chain ID in `AgentHubRegistry`; required for `POST /jobs` (`provider_id` in the body) |

`request_id` is generated by the API on creation. `registry_provider_id` is filled after the `ProviderRegistered` event (backend listener) once `registerProvider` is confirmed on-chain.

## Register a provider

`example/config/provider.json` no longer contains `provider_id`: the API generates a `bytes32` `request_id`.

`scripts/register-provider.ts` calls:

```ts
const result = await client.providers.create({
  name,
  description,
  owner_wallet,
  signer_wallet,
  payout_wallet,
  api_base_url,
  trust_level,
  service_type,
  input_schema,
  output_schema,
  price_usdc,
  max_concurrent_jobs,
  timeout_seconds,
});

// result.request_id  → API identifier (bytes32)
// result.transaction → registerProvider(signer, payoutWallet, price, workTimeout, metadataCommitment, expiresAt, registrationAttesterSignature)
```

The API creates the provider in the database with `status: REGISTERED`, computes a `metadataCommitment` from the metadata, signs the EIP-712 authorization server-side, then returns the `registerProvider` transaction. The on-chain `msg.sender` must be `owner_wallet`, and `signer_wallet` becomes the key authorized to sign `StartJobAuthorization` for that provider. If `signer_wallet` is omitted, the backend defaults to `owner_wallet`. If `SIGNER_WALLET_PK` and `RPC_URL` are set, the script signs and broadcasts the transaction with ethers. Otherwise, it only prints the prepared transaction.

After on-chain confirmation, the listener fills `registry_provider_id` and sets the provider to `ACTIVE`. The script waits for this link when the transaction was sent; otherwise fetch it via `npm run providers:list --prefix example` and put it in `example/config/job.json` under `provider_id`.

## Create a job

`scripts/create-job.ts` reads `example/config/job.json`. The `provider_id` field is the on-chain **`registry_provider_id`** (not the API `request_id`).

```ts
const transaction = await client.jobs.create(job);
```

The provider must exist in the API **and** have a `registry_provider_id` set (on-chain registration complete). The API returns a prepared transaction for `AgentHubEscrow.createJob`.

If `SIGNER_WALLET_PK` and `RPC_URL` are set, the script:

1. reads the provider price on-chain via `AgentHubRegistry.getProvider(provider_id)`,
2. checks `allowance(user_wallet, escrow)`,
3. sends `approve(escrow, price)` if allowance is insufficient,
4. signs and broadcasts `createJob`.

Otherwise, it only prints the prepared transaction (remember to approve the payment token manually before sending `createJob`).

## Start a job

`scripts/start-job.ts` reads `example/config/start-job.json`, then calls:

```ts
const authorization = await client.jobs.requestStartAuthorization(job_id, {
  expires_in_seconds,
});

const provider_signature = await signer.signTypedData(
  authorization.typed_data.domain,
  authorization.typed_data.types,
  authorization.typed_data.value
);

const started = await client.jobs.startJob(job_id, {
  provider_signature,
  expires_in_seconds,
});

const output = { text: `1 + 1 = ${1 + 1}` };

const finished = await client.jobs.finishJob(job_id, {
  output,
});
```

The `startJob` call returns `input` with relayed transaction metadata (`transaction_hash`, `relayer_address`, `block_number`, `gas_used`). The script then runs a trivial compute (`1 + 1`) and calls `job-finish` with `output`. `SIGNER_WALLET_PK` must match the `signer_wallet` registered for the provider.

## Accept a job

`scripts/accept-job.ts` reads `example/config/acceptance.json`, then calls:

```ts
const acceptanceRequest = await client.jobs.requestAcceptance(job_id, {
  output,
  expires_in_seconds,
});

const user_signature = await signer.signTypedData(
  acceptanceRequest.typed_data.domain,
  acceptanceRequest.typed_data.types,
  acceptanceRequest.typed_data.value
);

const accepted = await client.jobs.acceptance(job_id, {
  output,
  output_commitment: acceptanceRequest.settle_with_user_signature_args.output_commitment,
  expires_at: acceptanceRequest.settle_with_user_signature_args.expires_at,
  user_signature,
});
```

This matches the REST flow:

```text
POST {{baseUrl}}/jobs/{{jobId}}/acceptance-request
sign EIP-712 JobAcceptance
POST {{baseUrl}}/jobs/{{jobId}}/acceptance
```

`SIGNER_WALLET_PK` must match the job's `user_wallet`.

## Refund after timeout

Both endpoints take no body. They return a prepared transaction to sign and send from the wallet.

Direct HTTP call:

```bash
curl -X POST "$API_URL/jobs/$JOB_ID/refund-after-queue-timeout"
curl -X POST "$API_URL/jobs/$JOB_ID/refund-after-final-timeout"
```

With the SDK:

```ts
const queueRefundTx = await client.jobs.refundAfterQueueTimeout(jobId);
const finalRefundTx = await client.jobs.refundAfterFinalTimeout(jobId);
```

With the example scripts:

```bash
npm run refund:queue --prefix example -- <jobId>
npm run refund:final --prefix example -- <jobId>
```

You can also set `JOB_ID` in the environment and run the commands without arguments. If `SIGNER_WALLET_PK` and `RPC_URL` are set, the scripts broadcast the transaction; otherwise they only print the prepared transaction.
